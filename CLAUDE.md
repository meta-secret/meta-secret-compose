## Overview

Kotlin Multiplatform (KMM) mobile app targeting Android (Jetpack Compose) and iOS (UIKit + minimal Swift proxy). Integrates a prebuilt Rust native library (`meta-secret-lib`) over FFI for cryptography and secret sharing.

- **Android:** JNI via `libmetasecret_mobile.so`
- **iOS:** `metasecret-mobile.a` + `MetaSecretCoreLibrary-Bridging-Header.h`

## Build Commands

```bash
# Build
./gradlew build

# Install on Android device/emulator
./gradlew installDebug

# Format + lint Kotlin
./gradlew ktlintFormat detekt

# Run tests
./gradlew test

# Run a single test class
./gradlew test --tests "com.example.MyTestClass"
```

iOS builds use Xcode directly. The Kotlin framework is generated by Gradle and embedded into the Xcode project.

## Architecture

**MVVM with unidirectional data flow.** No `expect/actual` — all platform abstractions are interfaces.

### Module Structure

```
composeApp/src/
├── commonMain/   — domain, use-cases, models, interfaces, DI (KoinModules)
├── androidMain/  — Android adapters, JNI bridge, DI (PlatformModule.android)
└── iosMain/      — iOS adapters, Swift/ObjC bridge, DI (PlatformModule.ios)
iosApp/           — Xcode target (UIKit + SwiftBridge entrypoint)
```

### Layers

| Layer | Location | Rule |
|---|---|---|
| `ui/scenes/` | View + ViewModel per screen | No business logic; no FFI |
| `core/` | Domain, use-cases, interfaces | All platform interfaces declared here |
| Platform adapters | `androidMain/` / `iosMain/` | Implement core ports; no upward deps |
| `models/apiModels/` | JSON models from Rust lib | Mapped to `appInternalModels/` in core |

### FFI Boundary

`MetaSecretCoreInterface` is the **only** interface allowed to call FFI. Only `MetaSecretAppManager` may use it. FFI calls must always be wrapped in `withContext(Dispatchers.IO)`.

### Key Core Classes

- `MetaSecretAppManagerInterface` / `MetaSecretAppManager` — main façade over meta-secret-lib
- `MetaSecretSocketHandlerInterface` / `MetaSecretSocketHandler` — event sync
- `MetaSecretStateResolverInterface` / `MetaSecretStateResolver` — type-state helper (used only by VMs)

### DI (Koin)

- Android composition root: `commonModule + PlatformModule.android`
- iOS composition root (SwiftBridge): `commonModule + PlatformModule.ios`
- Every interface must have a Koin binding where its implementation lives
- Public APIs never expose `*Impl` types

### Interface Placement Policy

When adding a capability:
1. Define an **interface** in `core/commonMain`
2. If implementation is platform-agnostic → keep in `commonMain`, bind in `di/KoinModules`
3. If platform-specific / FFI → put in `androidMain` or `iosMain`, bind in `di/PlatformModule.android` / `.ios`
4. Public APIs expose interfaces only — no `*Impl` in signatures

### ViewModel Contract

```kotlin
interface CommonViewModel {
    val state: StateFlow<UiState>
    val navigationEvent: SharedFlow<NavigationEvent>  // optional
    fun handle(event: CommonViewModelEventsInterface)
}
```

All UI inputs go through `handle(event)`. Events are scene-specific sealed types implementing `CommonViewModelEventsInterface`. Internal state machines are private.

### Navigation

- **Android:** Navigation-Compose
- **iOS:** UIKit push/pop

## Code Style

- **No `!!`** — use explicit null checks or `requireNotNull` with a message
- **No `GlobalScope`** — use `viewModelScope` or injected scopes
- **No exceptions across module boundaries** — use sealed domain errors
- FFI/native errors are mapped once in `core`; UI receives sealed types only
- Callbacks/lambdas are always the last parameter
- Immutability first (`val`); functions ≈ up to 40 LOC
- **No comments in code — only code.**

### Naming

- Packages: `lowercase.with.dots`
- Types: `PascalCase`; members: `camelCase`; constants: `UPPER_SNAKE_CASE`
- Use-cases/ViewModels: `UseCase`, `ViewModel` suffixes
- Repositories: `XxxRepository` (interface) / `XxxRepositoryImpl` (impl, hidden behind DI)

### Logging Format

```kotlin
println("👤ProfileViewModel: ✅ User loaded")
println("👤ProfileViewModel: ❌ Failed to load user: $reason")
```

Same emoji used consistently within a file and its paired View/ViewModel. No PII or secrets in logs.

## Security

- iOS secrets/tokens → **Keychain** only
- Android secrets/tokens → **Keystore**-backed storage only
- `ui/` must never handle raw secrets; access only through `core/` interfaces
- Permissions limited to: biometrics, camera, push notifications

## Hard constraints
- Do not touch external native libraries.
- Do not change signing, provisioning, certificates, or Xcode team settings.

## Default repair workflow
1. Run the narrowest possible build first.
2. Classify the error into one category.
3. Produce a repair plan.
4. Wait for user confirmation.
5. Apply the smallest possible fix.
6. Re-run the relevant build.
7. If still failing, repeat with a new diagnosis.

## Priorities
1. Restore build with minimal changes.
2. Preserve architecture and existing conventions.
3. Prefer local fixes over broad refactors.
4. Explain uncertainty explicitly.

- Do not assume iOS Simulator is available or relevant.

- For iOS runtime issues, prefer a manual device-log workflow.
- Ask the user to run the app on a physical iPhone and provide:
    - crash message
    - Xcode console output
    - device log snippet
    - screenshot of runtime error if needed
- Treat runtime issues separately from build issues.
- Do not declare the problem solved until runtime verification is either:
    1. confirmed by the user, or
    2. explicitly deferred by the user.

## SOLID rules for this project
- Single Responsibility: a type must have one reason to change. Do not combine UI state, navigation, networking, persistence, and domain decisions in one class.
- Open/Closed: prefer extending behavior via new use-cases, adapters, or implementations instead of editing unrelated existing logic.
- Liskov Substitution: implementations of interfaces must preserve expected behavior and contracts.
- Interface Segregation: prefer narrow interfaces focused on one capability. Avoid large manager-style interfaces with unrelated responsibilities.
- Dependency Inversion: UI and platform layers must depend on abstractions from core, not concrete implementations.

## Architecture decision rules
Before writing code:
1. Identify the layer where the change belongs.
2. Check whether an existing interface or use-case already owns this behavior.
3. Prefer adding or extending a focused abstraction over putting logic into View, ViewModel, or platform bridge.
4. Keep Swift proxy code minimal; business logic must remain in shared/core layers unless clearly platform-specific.
5. Never place FFI calls outside the allowed FFI boundary.

## Forbidden architectural anti-patterns
- No business logic inside SwiftUI/UIView/UIKit views.
- No direct FFI usage outside MetaSecretCoreInterface and its allowed façade path.
- No platform-specific logic leaking into shared domain abstractions unless explicitly designed for that.
- No giant manager classes with mixed responsibilities.
- No direct dependency from UI to concrete data/network/native implementations.
- No cross-layer shortcuts that bypass core interfaces.
