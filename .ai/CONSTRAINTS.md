## 🚨 Core Constraints (Must Know)

### Vault Model
- **Vault** = trusted group of user's devices
- User starts with 1 device, can add more
- When >1 device: each secret splits via **Shamir Secret Sharing**
- Sharing scheme: **k = n - 1** (can lose 1 device safely)
  - 1 device → whole secret locally
  - 2 devices → 1-of-2 (both shares needed)
  - 3 devices → 2-of-3 (any 2 shares needed)
  - 4+ devices → same pattern (k = n-1)

### Device Master Key (DMK)
- Generated by Core, unique per device
- Must be saved on device in hardware-backed secure storage
  - **iOS:** Keychain (`kSecAttrSynchronizable: false`) — survives reinstall, no iCloud sync
  - **Android:** AES-256-GCM encrypted file in `noBackupFilesDir`, key in Android Keystore — does NOT survive reinstall (accepted behavior)
- Used to create AppManager and restore database
- NOT transmitted to server
- NOT a password, NOT a share, NOT Vault key
- NOT synced to cloud on either platform

### Server (What it does NOT do)
- ❌ Never stores secrets
- ❌ Never stores shares
- ❌ Never stores Device Master Key
- ❌ Never stores device database
- ✅ Only: signaling, message delivery, device communication

### Shamir Secret Sharing
- Each user secret splits **separately** (not one Vault key)
- Not used with 1 device (secret stored whole)
- Always k = n - 1
- Old shares invalid after resharing

## ⚡ Quick Reference

**Never allow:**
- ❌ Server storing secrets/shares/keys
- ❌ Vault Data Key usage
- ❌ Recovery Key usage
- ❌ Bypassing constraint validation (Stage 3.5)
- ❌ Deviations from k=n-1 schema
- ❌ Device removing itself
- ❌ Removing last device
- ❌ Cloud backup/sync of any app data (DMK, DB, keys)
- ❌ Storing DMK in external storage or plaintext

**Always require:**
- ✅ Approval for join/remove
- ✅ Resharing before removal
- ✅ DMK saved to hardware-backed secure storage (Keychain / Android Keystore)
- ✅ DB file deleted on clearAll (including files from old master keys)
- ✅ Biometry or PIN for operations
- ✅ Constraint validation before coding
